# C2 work-order decomposition: adversarial review verdicts

Four skeptics (session anchoring/lifecycle; session-scoped delivery; session
servability + the ifElse-gap claim; session confidentiality + executor
serving) attacked the C2 table and scout map against the design and the
landed C1 code; the synthesis judge re-verified every cited seam. 16 raw
findings deduped to 14 verdicts, all CONFIRMED or PLAUSIBLE, none refuted.
Headline: **C2.2's "meets the ifElse gap" is a false acceptance criterion**
(the red product-fixtures failure is a rank-independent space-scoped
map/ifElse write-surface defect, W2.15/W2.16 territory), and the
session-claim over-broadcast is **quadratic client compute** (sibling-session
fail-open reruns), not benign wasted delivery. The amendments below (CA1–CA14)
are owed into the plan's C2 table.

## Synthesis summary

Re-verified every cited seam directly in the code at HEAD (96bc48b81). No finding is REFUTED — all core code facts hold. After dedup, 14 verdicts: 5 CONFIRMED blockers, 6 CONFIRMED serious, 1 CONFIRMED minor, 2 notes (1 CONFIRMED, 1 PLAUSIBLE). The blockers cluster into three genuinely-unowned/miscited seams plus two runtime-correctness gaps: (1) session issuance-binding (server.ts:4076 hard-returns null), admission (engine.ts:8259), and commit-lane (engine.ts:7451) are all user-hardcoded and the scout mischaracterizes the engine guards as auto-passing — verified they require explicit edits; (2) scope-naming-link.ts hardcodes scope:\"user\" (lines 61/131) and is uncited by any WO while the emit side genuinely produces scope:\"session\" (pattern-binding.ts:274-279 + scope.ts) — the scout's 'already scope-agnostic/scope-general' is false and C2.2 cannot meet its own acceptance; (3) laneScopeKey (v2.ts:2140) has no broader-in-chain collapse, so a session lane reading a user input phantom-mis-keys it (verified resolveScopeKey ignores sessionId for user + laneOf drops the frame to the space lane); (4) the session-claim over-broadcast (#sessionAcceptsClaim server.ts:3299-3303) forces sibling-session fail-open reruns via the chain-keyed action index (v2.ts:1113/5827 → facade.ts:2184) — quadratic client compute, NOT 'wasted delivery'; (5) the push-side confidentiality of server-lane-authored session writes is unowned and the plan/scout contradict on the F6 dependency. EXPLICIT RULING on the four adopted defaults: NONE is unsound as a decision — deferring context-lattice-claims-v2, shipping computation-only, deferring the lane budget, and keeping one-Worker topology are all defensible. But two rest on false support: default #1's rationale ('over-broadcast harmless') is factually refuted by the verified sibling-rerun mechanism (Finding 5), and defaults #3+#4's shared safety net is illusory — default #4 reuses W2.9's non-latency parity bar for the §7 latency gate (contradicting the scout's own blocker #2 that an owner ms budget is required) and the fixed ≥3-lane floor is structurally blind to the hundreds-of-lanes serialization regime that the budget/topology deferrals exist to manage. ### [CONFIRMED / blocker] Over-broadcast session claims force sibling-session fail-open reruns (quadratic client compute), not 'wasted delivery'; default #1's rationale is false

*Work orders:* C2.1, C2.6

Full causal chain re-verified in code. (1) #sessionAcceptsClaim (server.ts:3299-3303) resolves the claim principal via principalOfSessionKey and accepts iff principal===session.principal — so session:alice:s1 is accepted by ALL of Alice's sessions. (2) The client stores every delivered claim (v2.ts:5559 claim.set). (3) The registered-action index is CHAIN-keyed: #executionActionsByKey via actionClaimChainMapKey (v2.ts:1113/1153), contextKey stripped. (4) On a sibling claim revoke/generation-replace (v2.ts:5581/5552), invalidateRegisteredExecutionActions (5822-5833) calls #executionActionsForClaimKey (chain key) and fires execution-claim-invalidation on EVERY registered action under that chain key with NO guard checking whether the receiving session's own claim is live — so session B's own registration of the shared action A is invalidated even though B's session:alice:s2 claim is still live. (5) facade.ts:2184-2189 turns that into prepareClaimedRerun + invalidateActionForHostWake = a fail-open rerun. This is spurious client recompute — the exact §6 duplicated-compute class the project removes — quadratic in same-principal concurrent sessions on every claim churn. Dependency graph permits the harm: C2.1 deps=[], C2.6 deps=[C2.1,C2.3], so session issuance can be dial-enabled before C2.6 narrows delivery. Default #1's conclusion (no v2 needed) still holds — the fix is C2.6's v1 predicate tightening — but its RATIONALE ('over-broadcast is wasted delivery, not a client race') is factually false.

### [CONFIRMED / serious] C2.4 read seam must return the GRANT's session id, not base.sessionId — a same-principal cross-session confidentiality trap
