### FB1 [CONFIRMED / blocker] F4b demotion against the real F3 server evicts the entire demoted closure and livelocks: the demoting watch.set response carries removes for every doc the dropped graph watches tracked, and the client wipes/evicts the just-registered members

*Found by:* panel. *Anchor:* `packages/memory/v2/server.ts:6386`. *Verifier votes:* refute-code=confirmed, repro=confirmed, impact=confirmed.

When the F4b client demotes its cold-boot graph watches to a docs membership watch via session.watch.set, the server computes the response sync with buildFullSync(old graph-tracked entities, new entities) — and a docs-only watch set contributes NO entities (groupedQueries skips the docs kind, server-sync.ts:132), so the frame carries removes for every previously graph-tracked doc. appendMemberUpserts (server.ts:6433) folds the freshly-seeded member upserts into the same frame but never suppresses the removes for docs that are current doc-set members. Empirically verified against the real server+client (loopback probe): watch.add(graph on of:x) then watch.set([docs of:x]) returns upserts [of:x@1] AND removes [of:x], and the client WatchView ends with entities [] — the same-frame remove-wins contract (client.ts:1840-1843, added by this same F3 commit) suppresses the seeded upsert and the remove passes the cross-frame guard (toSeq >= seq, client.ts:1857). On the runner side the demotion response is applied via applySessionSync (v2.ts:3299), whose remove loop calls evictHeldSpaceDocsSync when the doc-set surface is engaged (v2.ts:4573-4580): every demoted doc without pending writes is DELETED from #docs and its watch coverage cleared (v2.ts:3320-3344); docs WITH pending local writes instead get confirmed reset to seq 0 under pending replay (v2.ts:4582). The eviction then schedules a reconcile (v2.ts:4603) that re-registers a SHRUNKEN membership without the evicted docs, so the server drops the members too. Every subsequent cold re-pull (watch.add) repopulates session.entities, and the next reconcile's watch.set diffs them away again — a permanent pull → demote → evict livelock in which boot roots read as absent between cycles. This violates FA3/F4b's binding text ('membership registers BEFORE the overlapping graph watch is narrowed/dropped … a doc covered by both kinds is delivered once, in one frame, under one watermark' — the doc is instead REMOVED in that frame; the mandated one-shot/non-subscribing cold-boot evaluation mode was not built), and makes the F5 W2.9 parity gate structurally unpassable: the measurement protocol's step 1 requires 'F4b demotion complete' (implementation-plan.md:1795), but engaging the flags destroys the replica instead of reaching a steady doc-set surface. The tip measurement record (96bc48b81) confirms the flag-on path has never been engaged live.

**Failure scenario.** Enable EXPERIMENTAL_SERVER_PRIMARY_EXECUTION_DOC_SET_WATCH against a doc-set-advertising server (the F5 gate's step-1 configuration). Client cold-boots the default app: provider.sync(root) pulls the boot closure via a subscribing graph watch; the scheduled reconcile sends watch.set([docs watch]) to demote. The real server's response contains removes for the whole held closure alongside the member seeds; the runner evicts every held doc from #docs (UI notified with all values vanishing), membership shrinks to empty on the follow-up reconcile, every read re-pulls cold, and each re-pull is evicted again by the next demotion — steady state is never reached, session.watch.refresh traversal never retires, and any doc with pending local writes has its confirmed base reset to 0 mid-replay. Independently verified by a second live repro through the runner path: 'SET upserts: [of:child, of:root]; SET removes: [of:root]', entering the permanent pull -> demote -> evict thrash.

### FB2 [CONFIRMED / serious] W2.13 direct-builder path certifies callbacks that write captured cells as write-free (captures never reified, gate ranges over params only)

*Found by:* panel. *Anchor:* `packages/ts-transformers/src/transformers/schema-injection.ts:1479`. *Verifier votes:* refute-code=confirmed, repro=confirmed, impact=confirmed.

schedulerOptionsForComputationBuilder runs the capability analysis over the raw authored builder callback and mints `{ completeSchedulerScopeSummary: true }` from a params-only summary, without ever checking the callback is capture-free. The analysis records writes only for receivers resolvable to parameters (capability-analysis.ts:3145-3149 skips method-call tracking when the receiver is a free identifier; :3541-3552 builds `params` solely from `fn.parameters`), so a direct lift() whose body writes a captured cell — a pattern-scope cell or a module-scope data cell — produces a clean summary and receives the read-only/write-complete certificate with no `materializerWriteInputPaths`. Reproduced at tip via the test pipeline: `pattern(() => { const log = new Writable<string[]>([]); const f = lift((input: {msg: string; important: boolean}) => { if (input.important) log.push(input.msg); return input.msg.length; }); return f; })` emits `lift(cb, argSchema, resSchema, { completeSchedulerScopeSummary: true })` with the closure over `log` intact in valid output; module-scope `const state = new Writable(0); export const f = lift((input: {a: number}) => { state.set(input.a); return input.a; })` likewise certifies. The gate's own doc states the required invariant — 'every parameter has an empty writePaths and there are no captured-cell writes' (derive-scheduler-options.ts:22-23) — but the second conjunct is enforced nowhere on this path. The closure-strategy path is sound only by construction because ClosureTransformer reifies captures into the merged input param before analysis (the commit's own regression test proves that path emits `materializerWriteInputPaths: [["counter"]]`); W2.13 extends the same gate to callbacks that never went through capture reification, which is precisely where the params-only write proof is invalid. This is the attack the panel charter names for W2.12/W2.13 — 'can anything obtain a read-only certificate while actually writing' — and the answer is yes for the direct-builder (and, for module-scope cells, the no-capture lift-applied fallback) arm.

**Failure scenario.** An author writes a direct-builder lift inside a pattern (or at module scope with a module-scope data cell) whose callback conditionally writes the captured cell, e.g. `if (input.important) log.push(input.msg)`. The transformer stamps it `completeSchedulerScopeSummary: true` with an empty write surface. At runtime the runner registers a completeActionScopeSummary whose writes contain only the direct output (runner.ts:4496-4523), the executor classifies the action claim-ready (servability.ts:283-386), and the server claims it. On any run where the captured-cell write executes, the write falls outside every declared envelope and the claimed transaction is rejected by the write-coverage firewall (servability.ts:494-516, `dynamic-write-outside-static-surface` class), de-claiming the action back to client-primary. A genuinely-writing action is thus statically certified write-free — violating W2.12's 'keep every rejection for callbacks with writes' boundary and the certificate's stated soundness contract — producing claim/de-claim churn and misclassification rather than durable corruption only because the runtime write firewall backstops the false certificate.

**Evidence.** foundBy: W2.12+W2.13. schema-injection.ts:1445-1502 (analysis at :1479-1485, gate at :1486-1488, no capture-freeness check; the lift-applied fallback arm at :2516-2524 relies on CaptureCollector's capture notion, which excludes module-scope bindings); derive-scheduler-options.ts:74-81 (readOnly proof ranges only over summary.params) and :22-23 (unenforced 'no captured-cell writes' invariant); capability-analysis.ts:3145-3149 (cell mutator tracked only when receiver resolves via aliases seeded from parameters) and :3541-3552 (states for free-variable roots dropped from the summary). Reproduced at branch tip 96bc48b81 by running transformSource (packages/ts-transformers/test/utils.ts) over the two sources above: both emit `{ completeSchedulerScopeSummary: true }` with no materializerWriteInputPaths (pattern-scope output keeps the lift inside the pattern body with the closure over the cell intact). Contrast control at tip: the capture-merged computed regression test (test/scheduler-scope-certificate.test.ts:324-343) shows the closure path correctly emitting materializerWriteInputPaths [["counter"]] for the same authored intent. Direct builders emitted no certificate at the branch base (ed46f266e schema-injection.ts had no scheduler-options path for builder lifts), so this false-certificate class is introduced by this commit.

### FB3 [CONFIRMED / serious] W2.15a selector write envelope omits the minted result document — every real selector run fails the dynamic claim firewall

*Found by:* panel. *Anchor:* `packages/runner/src/runner.ts:5202`. *Verifier votes:* refute-code=confirmed, repro=confirmed, impact=confirmed.

The ServerBuiltinComputationDescriptor's write surface (writes: schedulingWrites, directOutputs: outputCells) covers only the node's direct root output spot, but ifElse/when/unless each mint and write a SECOND document — the `{ ifElse|when|unless: cause }` result cell (runtime.getCell(parentCell.space, { ifElse: cause }) at if-else.ts:57-62, written via setRawUntyped at if-else.ts:79, when.ts:46, unless.ts:46). Result: the summary classifies claim-ready statically, but dynamicActionTransactionUnservableReason (servability.ts:492-529) rejects every run that actually writes — including the very first run and every condition flip — with dynamic-write-outside-static-surface. Verified empirically against tip with the branch's own executor-router harness (createExecutorActionTransactionRouter, same shape as server-execution-product-fixtures.test.ts): a space-scoped pattern using all three selectors produced 6/6 attempts (cold + after-flip, all three builtins) that were static=claim-ready yet dynamicReason=dynamic-write-outside-static-surface. So the W2.15 deliverable — serving the selector cohort — can never serve any output-producing run; only literal no-op re-runs can settle under a claim. This contradicts the commit message's verification claim ('each writes only its result cell'; 'the fail-closed envelope bounds them exactly') and undermines the checked-off W2.15 success criterion ('ifElse observed claiming and settling in-run', implementation-plan.md:1620-1624). It also shows the C2 plan's attribution is wrong: implementation-plan.md:1909-1910/1934 records the lunch-poll/group-chat ifElse dynamic-write-outside-static-surface gap as a session-lane-rank problem to be met by C2.2, but the gap reproduces on a purely space-scoped selector, and neither lane-rank widening nor the §4 pair (which widens only across scope instances of the SAME doc id, servability.ts:484-491, 522-525) can cover a differently-id'd minted document. ### FB6 [CONFIRMED / serious] FA8 shrink broken for same-id watch.set replacement: a narrowed docs watch keeps delivering docs the session un-watched (reproduced live)

*Found by:* panel. *Anchor:* `packages/memory/v2/server.ts:6405`. *Verifier votes:* refute-code=confirmed, repro=confirmed, impact=confirmed.

watchSet's doc-set shrink computes droppedSources per WATCH ID only (server.ts:6396-6409): a docs watch whose id survives the replace keeps its `watch:<id>` source on every member it registered before, and #registerDocSetMembers (server.ts:6849) only ever ADDS. So `session.watch.set` with the same watch id and a smaller (or different-branch/scope) docs list never evicts the removed members — they stay in docSetMembers and trackedIds forever, keep waking the session, and keep fanning out deltas for documents the client explicitly un-watched. A later commit to of:y is still delivered to the session. ### FB7 [CONFIRMED / serious] FA4's 'next read misses and re-pulls' is broken on the production read path: eviction clears replica coverage but not the runtime/manager missing-doc kick latches, so an evicted doc's re-pull is deduped away and the client goes silently stale

*Found by:* panel. *Anchor:* `packages/runner/src/storage/v2.ts:3320`. *Verifier votes:* refute-code=confirmed, repro=confirmed, impact=confirmed.

evictHeldSpaceDocsSync deletes the #docs record, #watchedIds entry, and #watchSelectorTracker coverage — but the actual UI read path for an absent doc is traversal -> reportMissingLinkTarget -> Runtime.ensureLinkedDocLoaded, which is gated by TWO lifetime dedup latches that F4's eviction never clears: Runtime.missingDocLoadKicks (runtime.ts:1376,1399 — cleared only when a kick FAILS) and StorageManager #docPullKicks (v2.ts:978, consumed in shouldPullDoc at 1471-1489 — 'one entry per (space, scope, id) for the manager's lifetime'). Both latches are justified by the pre-F4 invariant that 'the first pull registers a server-side watch that keeps the doc flowing afterwards' (v2.ts:972-976) — the exact invariant F4's membership shrink retracts. So FA4's binding text ('any membership retraction evicts the doc from the replica in the same step so the next read misses and re-pulls') holds only through the bare provider.sync path the unit test uses; through the runtime read path the re-pull is suppressed and the reader's tracked read never fires. This also realizes FA4's explicitly forbidden 'stale-hit retry loop': a commit whose read set names the evicted doc conflicts, the doc-set catch-up for a non-member is empty (pinned by packages/memory/test/v2-feed-retirement-test.ts:381), and every retry re-reads the same absent/stale state because the pull is latched away.

**Failure scenario.** Doc D enters the replica via a missing-link kick (the common 'fresh-replica read asymmetry' path — or via any first eviction's recovery kick), populating both latches. Later an unlink retraction (graph-diff remove from a transient cold-discovery watch or mixed surface) evicts D. Another device re-links D under a member root: the root's new value is delivered (it is a member), the reader follows the link, the traversal read misses (record deleted), reportMissingLinkTarget fires, and ensureLinkedDocLoaded returns immediately at missingDocLoadKicks.has(key) (runtime.ts:1399) — no fetch is ever issued, D is not a member, and no graph watch survives demotion, so nothing ever delivers D.
