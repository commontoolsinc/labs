# Vouched Ingest Channel — The Grant: Final Implementation Plan

*Companion to `vouched-ingest-channel.md` (loom) and `vouched-ingest-channel-mint-design.md` (labs). Covers the ACL **grant** + webhook create→grant rework + minimal inspect/revoke, fitting the already-built ExternalIngest atom / split-mint / `custodyIngest` / location-ingest pieces. Every claim below was re-verified against the worktree at `/Users/alex/Code/worktrees/labs/vouched-ingest-channel` (branch `vouched-ingest-channel`).*

**Status:** plan (hardened after a 14-agent adversarial review: 7 subsystem mappers, 5 skeptic lenses, 9 blockers resolved) · **For:** Berni's ACL sign-off + the build · **Branch:** `vouched-ingest-channel`

## Why

The repo ships three half-built external-ingest edges — webhooks (`sendToStream` overwrites a transient stream, `webhooks.utils.ts:222-237`), OAuth, and Plaid (`custodyIngest.set`, `oauth2-common.utils.ts:184`, `plaid-oauth.utils.ts:195`) — and the new location beacon (`location-ingest.handlers.ts`). They all write external bytes into the fabric as the operator with no owner-granted, revocable authority gating the write. The committed location-ingest endpoint is worse than half-built: it is a **forge hole** — it trusts a request-supplied `auth.space` with zero binding to any channel and writes operator-signed, `ExternalIngest`-stamped bytes into *any* space DID a caller names (`location-ingest.handlers.ts:23-26` → `location-ingest.utils.ts:72-77`; `auth.space` is a free-form string, `location-ingest.routes.ts:13`). This PR closes that boundary at birth: an owner mints a revocable, space-scoped `WRITE` grant from their own authority, the operator's ingest write is gated against an **un-spoofable server-side channel registry** *and* the owner's grant, and revocation is effective at the very next POST. Because it is a trust boundary, the order of operations (who seeds the channel space, who holds standing, when the secret is released) is itself load-bearing and must be a tested invariant, not a UI convention.

## 1. The grant model, settled

These are the v1 answers — the shape that fits the code as built, with the deviations the review surfaced now stated plainly.

**Who owns a channel.** The **human owner** (the user's own client identity), never the operator. A channel is its own dedicated space; the owner becomes `OWNER` of `of:${channelSpace}` via the existing creator-seed seam (`#ensureCreatorSeeded`, `server.ts:830-855`). The toolshed operator stays *out* of ownership: it performs only the marked payload writes via `custodyIngest` (`as: identity`, `toolshed/index.ts:17-25`) and holds an explicit, scoped `WRITE` grant on each channel space (§4). The channel space contains **nothing but ingest trail cells** — no cross-tenant data, no owner secrets — because the presenter's WRITE is space-grained (see threat model below).

**How the channel space is provisioned — the false dichotomy is dropped.** The review (Berni lens) is correct that the draft's "anonymous `inSpace()` is categorically safer" claim was wrong. `anonymousSpaceName` returns `toURI(createRef(...))` — an entity URI (`of:...`), **not** a `did:` (`pattern.ts:655-658`). So `resolveSpaceNameSync` does not short-circuit (`isMemorySpaceDID` fails for `of:...`, `runtime.ts:1166`) and `resolveSpaceName` falls through to `createSession({spaceName: name})` → `Identity.fromPassphrase("common user").derive(name)` (`runtime.ts:1179-1185`, `session.ts:24-30`) — the **same** passphrase derivation the draft condemns. The cited "SECURITY INVARIANT" (`runtime.ts:1186-1191`) only means the runner won't *adopt* the derived key as a signer; it does **not** make the DID per-user-keyed.

Corrected rule: **the only collision-safe provisioning is a fresh, non-name-derived DID.** Any caller-named string (`spaceName`, or `inSpace("my-channel")`) collides cross-tenant. Two acceptable routes:
1. **`Identity.generate()` a fresh keypair**, address its `.did()` as the channel space. **The generated space key is used ONLY to derive the DID and is never persisted or used to sign anything** — generate and drop it in one function scope. If it leaked, it would be implicit `OWNER` forever (`server.ts:719-723`, `principal === space → OWNER`), outside the ACL.
2. **Anonymous `inSpace()`** (`pattern.ts:655-658`) — safe **only** because `frame.cause` is per-user-per-event unique (a cause-unique name), **not** because it bypasses `fromPassphrase`. Use only inside a pattern handler where no signer is ever materialized.

**Prefer route 1 uniformly** for the settings-UI path; it structurally removes the "discard the key" footgun only if the key is dropped immediately. **Never** `createSession` with a user-chosen `spaceName`.

**What the presenter credential actually is — and what it does NOT attenuate.** The presenter holds its own ed25519 `did:key` and self-signs an ordinary `session.open` whose `iss` is that DID; `verifySessionOpenAuthorization` returns that DID (`session-open-auth.ts:126-139`). The proposal's *"no DID to BE a principal"* is **not literally true** — the bearer secret **is** the private key; "bearer" is operational (key bytes handed to a third party), not "no-principal."

Crucially (review, security lens): **the presenter's `WRITE` grant authorizes space-wide writes to the channel space at the memory layer.** `#authorizeMessage` gates any non-ACL-doc write at `WRITE` (`server.ts:1467-1471`), and that is space-grained. "Append to one target only" is an **operator-endpoint convention** (`custodyIngest.append` into per-day trail cells, `location-ingest.utils.ts:70-78`), **not** a capability constraint. A presenter that bypasses the endpoint and talks to `/api/storage/memory` directly with its `did:key` can write/overwrite anything in that channel space, including forging or clobbering prior trail entries. This is acceptable **only because the channel is its own dedicated space** (blast radius = that channel's own data) — which is exactly why the channel space must hold nothing else.

**How the owner issues `{ [P]: WRITE }` — corrected on the value shape.** One ordinary `transact` with a `set` op at `id = of:${channelSpace}`. The **stored document** is `{ value: <ACL map> }` (what `#resolveCapability` reads via `document?.value`, `server.ts:725-726`); the raw double-nesting `{ value: { [owner]: OWNER, [presenter]: WRITE } }` appears **only** at the raw-transact test layer (`v2-server-acl-test.ts:105`). Through `ACLManager`, the author passes the **bare** ACL map `{...acl, [user]: cap}` and the cell/engine layer adds the `{ value: ... }` wrapper (`acl-manager.ts:36-49`: `#getCell().withTx(tx).set(acl)`). The provisioning test must therefore assert the double-nesting at the **Engine.read / graph.query** layer (as `v2-server-acl-test.ts:138-149` does), **not** on `ACLManager`'s argument — and must not double-wrap through `ACLManager`.

`ACLManager.get()` **throws on an empty/uninitialized ACL doc** (`acl-manager.ts:25-27`: `Object.keys(aclData).length === 0`), so the channel space must already carry a seeded `{ [owner]: OWNER }` doc before the first `set()`. This is a *guard*, not the *enforcer* of ordering — see §1's enforce decision and §2 Step 0.

**How enforce is achieved — per-space `#aclModeFor(space)`, and it is the load-bearing security pivot, not a mechanical change.** `#aclMode()` is server-global today (`server.ts:702-704`, fed once from `env.MEMORY_ACL_MODE`, default `off`; wired through `memory.ts:38`). Three options:
- ✗ **Global flip to `enforce`** — fail-closed cliff. Every existing data-bearing space (`serverSeq>0`, no ACL doc) starts denying (`#resolveCapability` → `null` → deny, `server.ts:725-732`); `#ensureCreatorSeeded` only seeds `serverSeq===0` spaces (`server.ts:838`). **Additionally** (review, enforce lens): under a global flip the **toolshed operator runtime itself** opens arbitrary user spaces (pattern execution, OAuth refresh) and is **denied at `session.open` READ** (`server.ts:1360-1366`) on every pre-existing user space where it is neither the space DID, a service DID, nor ACL-granted — the operator's own server-side flows fail closed fleet-wide. This reinforces the rejection.
- ✗ **Second pinned-enforce Server instance** — `standalone.ts:40-53` makes it mechanically trivial. Rejected on **operational disproportion and split-brain**, *not* a missing routing capability: the storage client **does** have per-space host routing (`spaceHostMap`, `v2.ts:449-457`; dynamic `registerSpaceHost`, `v2.ts:663-683`; `v2-remote-session.ts:58-73`). The draft's "no space→host routing" claim was factually wrong and is retracted. The real reason: two Server instances over distinct stores (the toolshed runs a module singleton over one `memoryHost`, `memory.ts:34`, `index.ts:24`) split a space's engine/seeding/`#aclCapabilities` state per host with no reconciliation — "a space lives in exactly one instance" becomes a **correctness invariant whose violation silently forks a space's ACL doc**, not a routing nicety.
- ✓ **Make `#aclMode()` per-space.** `#aclMode` is *defined* at `server.ts:702` and *called* at four sites: `#authorizeMessage:762,769`, `#revokeDeauthorizedSessions:799`, `#ensureCreatorSeeded:835`. All four have `space` reachable (params on `#authorizeMessage`; locals on the two private helpers). Thread `#aclModeFor(space)` returning `enforce` when `space` is a registered ingest-channel space, else the global floor (`off`). This is a **1-signature + 4-call-site change** (not five; `:702` is the definition; `#capabilityFor`/`#aclCapabilities` do not gate on mode). **Flag explicitly for Berni:** gating `#ensureCreatorSeeded` on `#aclModeFor(space)` is the *same switch* that arms the seeding race (§9 risk 5) — seeding-under-enforce and the provisioning-ordering guard are **one decision**, not two.

**The membership source is security-critical and must be pinned before Step 1 lands** (§7 Q2, §9 risk 1). Derive enforce-eligibility from a **registry doc in a SERVICE-owned space** — a space whose DID the operator holds OWNER on via `MEMORY_SERVICE_DIDS` — so only the operator (writing at provisioning) can add a channel. **Never** from a doc in the channel space itself (client-writable) and **never** from a request field. A client-settable flag would let an attacker opt their own space *out* of enforce (disabling revocation) or a victim space *in* (DoS).

**The honest enforcement story — the central design fork, named.** In the operator-mediated ingest path the byte-write commits as the **operator** DID, not the presenter (`custody-ingest.ts:58` `editWithRetry` on the operator runtime built `as: identity`, `index.ts:17-25`). So at the memory server `#authorizeMessage` checks the **operator's** capability (`server.ts:1467-1470`), never the presenter's; the owner's `{ [P]: WRITE }` does **not** gate the byte-write, and `#revokeDeauthorizedSessions` (`server.ts:794-823`) tears down nothing — the presenter holds no memory session. This **directly contradicts** the proposal's load-bearing claim *"v0 enforcement is real — the ACL check — not an app-layer interim"* (`vouched-ingest-channel.md:25`).

**Resolution (Option A, chosen):** for v1 the **app-layer membership+WRITE recheck IS the enforcement mechanism** for the operator-mediated path; the ACL fact is the *source of truth*, re-read per-POST. Per-space `enforce` is **defense-in-depth** that gates the *operator's own commit* (and any real presenter session, the deferred `prf` form). The proposal language must be downgraded from "real ACL check" to **"the ACL fact is the authoritative grant, re-read on every ingest; per-space enforce hardens the operator commit."** Option B (open a presenter-principal memory session so `#authorizeMessage` gates directly) is rejected because it **bypasses the operator-side split-mint** — `stampExternalIngest` is module-private to the operator runtime (a deliberate anti-forge-oracle `WeakMap`, `external-ingest.ts:38-67`), so a presenter-authored commit could not carry the trusted mark. The tension between "real ACL enforcement" and "operator-minted provenance" is fundamental; we resolve it in favor of the mint and make the app-layer check the enforcement.

**What stays deferred** (§8): the screening caveat `prompt-injection-risk-unscreened` (its own labs PR — never minted today); the `prf`/`Delegation[]` chain (`[]`); `aud`/audience enforcement (recorded-not-enforced; federation PR5); per-target/per-cell ACL.

## 2. Sequenced implementation steps

Small, independently-committable. The dependency graph is drawn honestly (review, scope lens): **Step 0 → 1 → 2 → 3 → 4 → 7 form one gated core** behind Berni's sign-off and the membership-source decision; **Layer A (Step 6) and Inspect (Step 5) are the only independently-landable slices.**

**Step 0 — The provisioning state machine (the crux, written as an ordered, tested sequence).** *(Berni? — membership source + seeding guard.)*

This replaces the draft's scattered risk notes. The exact order, with each precondition:

1. **Register the channel space in the enforce set** — operator writes the channel's DID into the **service-owned registry doc** (a doc in a `MEMORY_SERVICE_DIDS` space). This must happen **before** the first owner open, because `#ensureCreatorSeeded` gates on `#aclModeFor(space)` (`server.ts:835`) and only seeds under non-`off` mode. *Membership cannot depend on a doc written inside the channel space.*
2. **Owner opens the channel space** → `#ensureCreatorSeeded` writes `{ [owner]: OWNER }` (`server.ts:838-852`). The server-side seeding guard (Step 1b below) ensures only the **registered owner** can be seeded for an enforce-set space.
3. **Owner authors the grant atomically-enough**: `ACLManager.set(presenterDID, "WRITE")` **and** `ACLManager.set(operatorDID, "WRITE")` on the channel space, confirmed committed (`storageManager.synced()`, `acl-manager.ts:51`).
4. **Only then** hand out the secret (presenter private key bytes + endpoint URL + channel space DID) and mark the channel enforce-eligible to ingest.

**Why each guard exists — the two seeding-race authority inversions, both confirmed in code:**
- **Presenter wins the seed** (`server.ts:1359`: `#ensureCreatorSeeded` fires for *any* principal opening a `serverSeq===0` space, *before* `#authorizeMessage`). The presenter holds a working `session.open` for `auth.space = channel space`; if it opens before provisioning completes, it is seeded `{ [presenter]: OWNER }` and locks the owner out.
- **Operator wins the seed** (review, integrator + enforce lenses — *worse* than the draft's risk 5, which only guarded the presenter). With the default empty `MEMORY_SERVICE_DIDS` (`env.ts:206`), the operator is **not** a service principal (`server.ts:706-708`), so it does **not** hold implicit OWNER. If an ingest POST arrives before the owner seeds and the operator's `custodyIngest` write (`editWithRetry`, `custody-ingest.ts:58`) is the first open of the `serverSeq===0` channel space, the server seeds `{ [operator]: OWNER }` — the operator silently owns the user's channel. Authority inversion.

**The hard fix (ask Berni — Step 1b):** make seeding a **server-side guard**, not sequencing discipline. `#ensureCreatorSeeded` for an enforce-set space must consult the same un-spoofable registry and **seed only the registered owner**; any other principal (presenter or operator) opening that space before it is seeded is **not** seeded as OWNER. Belt-and-suspenders at the endpoint: **refuse to ingest to a channel space whose ACL doc is absent or lacks `{ [owner]: OWNER }`** (Step 3). This makes "owner-opens-first" a tested invariant, not a UI convention.
- **Test (negative, required):** operator's first-ever open/write of a registered-but-unseeded channel space does **NOT** seed it OWNER; presenter open before provisioning is denied/not-seeded.

**Step 1 — Per-space enforce resolver + seeding guard in the memory server.** *(Berni? — his machinery, the load-bearing net-new ACL work.)*
- `packages/memory/v2/server.ts`: `#aclMode()` → `#aclModeFor(space)` returning `enforce` when `space` is in the server-side ingest-channel set, else `this.options.acl?.mode ?? "off"`. Update the four call sites (`:762,769,799,835`).
- Add the registry-doc source on the `Server` `acl` option (read from a service-owned space; **Berni Q2** pins marker-doc-in-service-space vs. equivalent). Same un-spoofable set drives Step 1b's seeding guard.
- **Step 1b:** `#ensureCreatorSeeded` consults the registry: for an enforce-set space, seed **only** the registered owner.
- **Test** (`v2-server-acl-test.ts`, new cases mirroring `:127-219`, `:362-433`): enforce-set space denies a non-granted principal while a sibling stays `off` and allows; granted presenter writes; **the seeding guard denies operator/presenter self-seed**; `#ensureCreatorSeeded` still seeds the registered owner on first open. The harness already constructs `mode:"enforce"` servers in-test (`createAclServer({mode:"enforce"})`, `:128-130`) — no global flag needed.

**Step 2 — Channel-space provisioning + owner-grant from a runtime-bearing client surface.** *(Depends on Step 0/1.)*
- **Architectural home — resolved (review, completeness lens).** `cf-webhook` is a Lit leaf that imports `CellHandle` from `@commonfabric/runtime-client` (`cf-webhook.ts:3,48-49`) and reaches toolshed via raw `fetch("/api/webhooks")` (`cf-webhook.ts:116`). It has **no `Runtime`, no `Identity`, no `ACLManager`** — and `ACLManager` is exported only from `@commonfabric/runner` (`runner/src/index.ts:180`), needing a full `Runtime` with `editWithRetry` (`acl-manager.ts:47`), which the browser `RuntimeClient` does not expose (it offers `getCell`/`getCellFromRef`, `runtime-client.ts:130,139`, not `editWithRetry`). The shell's `ACLView` is **disabled during the RuntimeClient migration** (`shell/src/views/ACLView.ts:11,160,175,286`). **The grant flow must therefore live in a runtime-bearing surface, not `cf-webhook`.** The proven, runtime+identity-bearing path today is the **CLI** (`packages/cli/lib/acl.ts:21,44,65-78` constructs a `Runtime` from a loaded identity and calls `ACLManager.set/remove`). For v1, build the grant helper as a runner-side module callable from that surface; `cf-webhook` becomes a thin presenter that *invokes* it, not the place that drives `ACLManager`. **Berni/team decision (§7):** confirm the v1 client surface (CLI-driven settings command vs. unblocking a shell runner path); the plan must not both reject a toolshed endpoint *and* lack a working client path. If only a server-side runtime is available short-term, the helper runs operator-side **but still authored under the owner's identity** (not the operator's) — never a soft caller-provided ownership check.
- Sequence: Step 0 (1)→(4), using `ACLManager` on the channel space.
- **Persist a channel registry record** (name, target, presenter DID, createdAt) in the **owner's own space** so "list my channels" reads the owner's space. This record is **distinct** from the service-owned enforce-membership registry (Step 1) — the owner record is for display; the service record is the un-spoofable enforce source.
- **Test:** provision → owner is OWNER of `of:${channelSpace}` (assert via Engine.read/graph.query, **not** ACLManager arg) → grant lands `{ [owner]:OWNER, [presenter]:WRITE, [operator]:WRITE }`; `set`-before-seed throws (`acl-manager.ts:25-27`); presenter/operator-wins-seed denied (Step 1b).

**Step 3 — The endpoint gate: membership FIRST, then presenter-WRITE, then seed-check (the highest-risk deliverable, fixes the committed forge hole).**
- The committed endpoint authenticates the presenter (`verifySessionOpenAuthorization`, `location-ingest.handlers.ts:16`) but then writes to the request-supplied `auth.space` (`:25`) with **zero binding to any channel registry**. A `did:key` holder can self-sign a valid `session.open` for **any** `sub` (the signature proves control of `iss`, never a space relationship — `session-open-auth.ts:90-100`). So any party who can mint a `did:key` deposits operator-signed, `ExternalIngest`-stamped bytes into any space they name. **This is the forge hole the feature exists to close, open in committed code.**
- **A single shared helper, invoked on EVERY operator ingest write, unconditional of server mode**, in this order:
  1. **`auth.space ∈ server-side channel registry`** (the same un-spoofable set as Step 1 / Berni Q2). *Load-bearing gate.* The draft's "read the channel ACL" does **not** close the hole alone: an attacker-chosen fresh space has no ACL doc, and under `mode:off` (the endpoint's default today) `#authorizeMessage` short-circuits to allow (`server.ts:762`).
  2. **Channel ACL doc present and names `{ [owner]: OWNER }`** (defense-in-depth vs. operator-winning-the-seed; refuse if absent).
  3. **Presenter `P` holds `WRITE`** on `of:${channelSpace}` — the *revocability* gate.
  4. Reject `"*"`/`ANYONE_USER` over-grant for ingest (`server.ts:732`).
- **Freshness (review, enforce lens — required):** the check must **force a fresh read** of `of:${channelSpace}` on every POST, not serve a cached ACL cell. Because the toolshed memory server is in-process (`memory.ts:34`), read the **memory server's engine directly** (server-side `Engine.read({ id: aclDocId(space) })`, the same path `#resolveCapability` uses, `server.ts:725`) rather than round-tripping a cached operator-runtime cell. If the read cannot be made synchronously fresh against the same engine the owner just wrote, the "next ingest" guarantee is false.
- **Reuse, don't re-derive (review, integrator minor):** export the resolution predicate from `@commonfabric/memory` (a pure `isCapable(resolveCapability(aclValue, principal), "WRITE")` built on the existing `isACL`/`ANYONE_USER`, `acl.ts`), and have the toolshed helper call it — **one source of truth** for "does P hold WRITE here," so the `*`-over-grant guard lives in one place.
- Files: new `packages/toolshed/lib/channel-acl.ts` — `assertIngestAuthorized(memoryServer, channelSpace, presenter)` — called from `location-ingest.handlers.ts` (before `appendLocationPoints`) and the reworked webhook ingest (Step 6); `401`/`403` otherwise.
- **Require `exp` on the ingest `session.open`** (review, two lenses — see §9 risk 6): reject an open whose invocation lacks `exp`, because server-side `exp` enforcement is **presence-gated** (`session-open-auth.ts:115`: `if (invocation.exp !== undefined)`). An external presenter that omits `exp` gets an **unbounded** replay window. Pass a max-age / require `exp` at this edge.
- **Test:** grant present → 200; owner `ACLManager.remove(P)` on a different runtime → very next POST **401** with no sleep/poll; unregistered `auth.space` → 401 even under `mode:off`; missing-`exp` open → 401.

**Step 4 — Operator authorization under enforce (pin it; the claim "not wired" was wrong).**
- Correction (review, three lenses): `MEMORY_SERVICE_DIDS` **is fully wired** — `env.ts:206` (default `""`) → `memory.ts:38-42` (split/trim/filter into `serviceDids`) → `server.ts:706-708` `#isServicePrincipal` → implicit OWNER on **every** space (`server.ts:719-723`). The value is *unset by deploy*, not the wiring. Adding the operator DID is a one-env-var change granting implicit OWNER fleet-wide — exactly the over-broad grant to avoid, and it would also make `#ensureCreatorSeeded` **skip** the operator (`server.ts:837`), changing the seeding behavior the tests assume.
- **Decision: grant the operator scoped `WRITE` on each channel space at provisioning** (authored in Step 0/Step 2 alongside the presenter grant). This is **required** under per-space enforce: without it the operator's own `custodyIngest` commit is denied (`isCapable` fails for the operator with no grant, `server.ts:765`). Co-order the enforce-membership flip and the operator-WRITE grant transactionally at provisioning, else the first operator commit after a space joins the enforce set is denied.
- **Berni/deploy ask (§7):** confirm the **Loom** toolshed deploy's actual `MEMORY_SERVICE_DIDS`, not just the labs default. If Loom already lists the operator as a service DID, the per-channel operator grant is dead code and the seeding tests are wrong — pin one model.
- **Test:** enforce server, operator-as-non-service, channel grants operator WRITE → `custodyIngest.append` succeeds; without it → denied.

**Step 5 — Inspect (minimal-viable, channel-type-agnostic, §5).** Net-new read-only panel; no server changes. Independent of Steps 0–4.

**Step 6 — Webhook ingest stream→trail retrofit, Layer A (§3).** Independent of enforce; lands in parallel.

**Step 7 — Webhook create→grant + cf-webhook UI rework, Layer B (§3).** Depends on Step 2 (grant helper) and Step 1 (enforce). For classic (non-DID) webhooks, revocation is registry-disable, **not** ACL — stated plainly (§3, §4).

## 3. The webhook rework

Two clearly separated layers; never dependent on a global enforce flip.

**Layer A — ingest stream→durable-trail contract change (low-risk, no enforce dependency).**
- **The line that changes:** `webhooks.handlers.ts:145` `sendToStream(...)` → `custodyIngest.append(trailCell, payload, channel)`. Retire `sendToStream` (`webhooks.utils.ts:222-237`: `streamCell.withTx(tx).send()` — transient fan-out, no provenance, not accumulating).
- **Target cell shape flips** from `asCell:['stream']` (`Stream<JSONValue>`) to a **list cell** (`JSONValue[]`), mirroring location-ingest's per-day trail (`dayCellCause`, `location-ingest.utils.ts:44-48`). **Bound growth** (cap length or day-key cells) — the stream never accumulated.
- **MUST route through `custodyIngest.append`**, not a hand-rolled `[...arr, x]` set — the dedicated mint pass anchors at the declared `target` path (`external-ingest.ts:24-36`, mint at `prepare.ts:3587`); a naive set would stamp a whole-payload digest on the wrong slot (mint-design Finding 2).
- **Existing-registration migration — explicit subsection (review, integrator blocker).** Today's webhooks live as toolshed service-space registrations keyed by `secretHash`, `createdBy: space` (`webhooks.utils.ts:21-29`, `handlers.ts:66-77`), authenticate with `Bearer whsec_` (`handlers.ts:107-132`). Before landing Layer A: (1) confirm whether prod/Loom has live `whsec_` registrations at all; (2) if yes, decide whether a cell previously provisioned as `asCell:['stream']` is backward-compatible with a list write, or whether existing registrations must be **re-created** — behavior on the schema mismatch (an existing webhook's next POST hitting `custodyIngest.append` against a stream-shaped cell) is otherwise unspecified, do not ship Layer A until decided; (3) **keep the `Bearer whsec_` path and the toolshed registry as the LIVE classic-webhook auth** — do **not** call it "vestigial". Layer B's ACL grant is for `session.open` presenters, and a GitHub/Stripe sender **cannot** self-sign (`handlers.ts:106-135`). The classic webhook does **not** migrate to the grant model.
- **Vendored-consumer blast radius (review, integrator major).** In labs the only `$inbox`/`cf-webhook` references are `ui/v2/index.ts` (registration), `cf-webhook/index.ts`, `webhook-test.tsx`, `jsx.d.ts` (type) — no catalog story, no home-schema. But `cf-webhook`'s **public prop contract** is `$inbox: Stream<JSONValue>` (`cf-webhook.ts:28-29,48`; `webhook-test.tsx:28,47`), and the integrator's target is **Loom**, which vendors labs. Any Loom pattern wiring a `Stream<JSONValue>` to `$inbox` and registering a per-event handler **silently stops firing** when the target flips to a list cell (a durable list never dispatches handler events). **Before landing:** grep the Loom checkout (and home-schemas as vendored there) for `$inbox` / `Stream<JSONValue>` / `cf-webhook`; enumerate consumers. Either (a) version the prop (`$trail` list vs. legacy `$inbox` stream) so existing stream consumers don't go dead, or (b) have the operator fan **both** a durable-trail append and a stream send during a deprecation window. "No in-repo consumer" is not "no consumer" for a vendored component.

## 4. Revocation & re-scope

- **Revoke (DID presenter)** = owner `ACLManager.remove(presenterDID)` (`acl-manager.ts:41-44`: read current ACL, drop `P`, write the **full reduced bare map**). The server's `commitTouchesAclDoc` (`server.ts:1463`) detects it, `#invalidateAclCapabilities` runs (`server.ts:1529`), and `#revokeDeauthorizedSessions` (enforce-only, `server.ts:794-823`) tears down any *real* presenter memory session.
- **The wiring caveat made the headline (review, two lenses):** in the operator-mediated path the presenter holds **no** memory session, so server teardown is **inert** for ingest. The **only** thing making "revoke → next write 401" true for ingest is the **Step 3 app-layer membership+WRITE recheck** (forced-fresh read). Therefore the **load-bearing acceptance test** POSTs to the location-ingest endpoint with a real operator runtime: grant → 200; owner `ACLManager.remove(P)` on a different runtime → very next POST 401, no sleep/poll. `v2-server-acl-test.ts:362-433` is **demoted** to "reference for the holds-a-real-session case (the deferred `prf` form)" — it does **not** exercise the operator path and must not be cited as proof of the ingest acceptance.
- **Re-scope / narrow target** = coarse by construction (channel-as-space; ladder is `READ<WRITE<OWNER`). Narrow = revoke `P` on the old channel space + grant `P` on a new, narrower channel space. Two `ACLManager` calls.
- **Rotate the secret** = present a new DID: `ACLManager.set(P_new,"WRITE")` + `ACLManager.remove(P_old)`. Two ACL writes; the UI presents them as a composition so a failed half can't strand a stale grant.
- **Concurrent-edit safety (review, security minor).** `ACLManager.set/remove` do a **read-modify-WRITE of the full map** with no concurrency guard (`acl-manager.ts:36-52`). Two concurrent owner edits can lose an update — last-writer-wins on the whole map. The mitigation already exists: `#write` goes through `editWithRetry` (`acl-manager.ts:47`), which detects commit-time read conflicts and retries rather than clobbering. **Verify `ACLManager.#write` surfaces `ConflictError` to the caller; add a concurrent set+remove test.**
- **Races already correct (do not re-implement):** owner self-removal still gets its commit response (`writerSessionId` carve-out, `server.ts:807-814`); presenter mid-transact serialized per-connection; cross-process `#aclCapabilities` staleness — per-Server-instance, invalidated only by that instance's ACL-touching commit (`server.ts:1529`) — holds under the single in-process toolshed singleton (`memory.ts:34`); flag as a latent multi-process hazard (§9 risk 7).

## 7. The Berni / deploy ask

**Q1 (the proposal's single explicit ask, `vouched-ingest-channel.md:57`):** Is a **space-grained `WRITE` ACL grant** — owner-written `{ [P]: "WRITE" }` at `of:${channelSpace}`, channel-as-its-own-space, presenter authenticated by the **unchanged** `verifySessionOpenAuthorization` (`prf` `[]`, `aud` unset) — the sanctioned way to express a revocable durable-append grant, with per-target ACL and the `prf`/`Delegation[]` chain rightly reserved? If yes, nothing here is interim: it is the minimal *real* form.

**Q2 (the plan's added mechanism — flagged as net-new, not smuggled under Q1):** Per-space enforce via `#aclModeFor(space)` is a **plan-proposed mechanism the proposal does not ask for**; we present it with our recommended default rather than as a co-equal blessing. Two parts: (1) Is gating `#aclModeFor(space)` (ingest-channel spaces enforce; others stay on the global `off` floor) the right shape vs. the global flip the surface text implies? (2) **What is the un-spoofable server-side source of the enforce-space set?** Recommended default: a **registry doc in a SERVICE-owned space** (only the operator, holding service-DID OWNER there, can add a channel; populated by provisioning). **And** will `#ensureCreatorSeeded` accept a server-side **seeding guard** (seed only the registered owner for an enforce-set space)? This is security-critical: a client-settable flag lets an attacker opt their space out of enforce (kill revocation) or a victim space in (DoS); an unguarded seed lets the operator or presenter capture ownership.

**Deploy item (blocking for Loom vendoring, not for labs build):** confirm the **Loom** toolshed deploy's actual `MEMORY_SERVICE_DIDS`. If the operator is already a service DID there, the per-channel operator-WRITE grant is dead code and the seeding tests are wrong — pin one model.

**For awareness (non-blocking):** the proposal's *"no DID to BE a principal"* is not literally true — the presenter has a real `did:key` (the bearer secret **is** its private key), and its space-grained `WRITE` authorizes whole-channel-space writes, not just append (§1, threat model §8). Reconcile the language rather than weaken the built model.

## 8. Out of scope / threat model

**Out of scope (this PR must NOT include):**
- Screening caveat `prompt-injection-risk-unscreened` propagation (its own labs PR; never minted today; the "unscreened" badge rides it).
- `prf`/`Delegation[]` chain (`[]`; the deferred maximal form).
- `aud`/audience enforcement (recorded-not-enforced; federation PR5). **But** payload-binding and `exp`-requirement at the ingest edge are **in scope** (§3 Step 3, §9 risk 6) — they are not the federation work.
- Per-target/per-cell ACL (scope is the space boundary; ladder `READ<WRITE<OWNER`).
- A second memory Server / space→host routing (rejected for split-brain).
- Operator-blind ingest (v1 is operator-trusted).
- Per-element historical digests in inspect (latest-ingest only).

**Threat model (v1, stated honestly):**
- **Operator-trusted mint.** `as: identity` sees plaintext; the split-mint protects the *mark*, not the *bytes* (`custody-ingest.ts:28-29`). `stampExternalIngest` is module-private to the operator runtime (`external-ingest.ts:38-67`), so a compromised/malicious toolshed can mint arbitrary "verified channel" provenance on arbitrary bytes — **the mark's trust root IS the operator.** The `ExternalIngest` mark attests *"the operator asserts this arrived via channel X,"* **not** a cryptographic third-party attestation (that is the deferred `aud`/attestation work, PR5). The inspect badge must **not** imply non-repudiation by the external service.
- **Leaked presenter key = full write of the channel space until revoke.** The `WRITE` grant is space-grained (`server.ts:1467-1470`); a presenter can bypass the endpoint, talk to `/api/storage/memory` directly, and write/overwrite/forge any trail entry in that channel space. Acceptable **only** because the channel is its own dedicated space (blast radius = that channel's own data) — so the channel space must contain nothing but ingest trail cells.
- **Replay (single host, today).** A captured `session.open` is bound to `sub` + session descriptor, **not** to the `points` payload (`session-open-auth.ts:90-100`); within TTL an on-path/log-reading adversary can replay the same auth blob with a **different** payload to the same channel. `aud` unset means it also replays to any host serving the endpoint for that space. Mitigations in this PR: **require `exp`** at the edge (close the unbounded-window hole); recommend **binding a payload digest/nonce** into the signed invocation as the stated hardening; treat TLS + a short `exp` as security parameters. Do not silently defer payload-substitution to PR5.

## 9. Open risks & decisions (with recommended defaults)

1. **Enforce-space membership source** *(Berni Q2 — must resolve before Step 1 lands).* **Default:** registry doc in a SERVICE-owned space, written at provisioning; never a doc in the channel space, never a request field.
2. **Operator authorization under enforce** *(Step 4).* **Default:** scoped per-channel operator `WRITE` grant at provisioning — **required** under enforce or the operator's own commit is denied (`server.ts:765`). `MEMORY_SERVICE_DIDS` is **wired** (`memory.ts:38`), default empty; adding the operator grants implicit OWNER fleet-wide (`server.ts:719-723`) — avoid. Confirm Loom's deploy value.
3. **Channel-space DID derivation** *(Step 0/2).* **Default:** `Identity.generate()`, key dropped in-scope, never persisted/used to sign. `inSpace()` only inside a pattern handler and only because the cause is unique — **not** because it bypasses `fromPassphrase` (`pattern.ts:658`→`runtime.ts:1179`→`session.ts:25`). Never `spaceName`/named `inSpace`.
4. **Enforcement mechanism for the operator path** *(the central fork, §1).* **Default:** app-layer membership+WRITE recheck IS the v1 enforcement (Option A); per-space enforce is defense-in-depth gating the operator commit. Proposal language downgraded from "real ACL check" to "ACL fact is authoritative, re-read per-POST." Option B (presenter-principal session) rejected — bypasses the operator-side mint (`external-ingest.ts:38-67`).
5. **Seeding-race authority inversion** *(both directions; Step 0/Step 1b).* **Default:** server-side seeding guard (seed only the registered owner for an enforce-set space) + endpoint refusal to ingest to an unseeded channel space. Hard, tested invariant — **not** sequencing discipline. Confirmed: `#ensureCreatorSeeded` fires for any principal before `#authorizeMessage` (`server.ts:1359`), seeds `serverSeq===0` (`:838`).
6. **`session.open` replay** *(in scope, partly hardened now).* `exp` enforcement is **presence-gated** (`session-open-auth.ts:115`); an external presenter omitting `exp` gets an **unbounded** window (runner-stamped opens are ~300s + 120s skew, `v2-remote-session.ts:43,207`, `session-open-auth.ts:68,121`). **Default:** require `exp` at the ingest edge; recommend payload-digest/nonce binding; document the payload-substitution risk.
7. **Multi-process cache staleness** *(latent).* Revocation immediacy assumes the single in-process toolshed singleton (`memory.ts:34`). Flag if toolshed ever runs multi-process.
8. **`*`/ANYONE over-grant** *(guardrail).* Never emit `"*"` for ingest — `#resolveCapability` falls back to `byPrincipal[ANYONE_USER]` (`server.ts:732`), so any authenticated DID inherits it. Enforce in the **shared** predicate (§3), one place.
9. **OAuth/Plaid user-space incompatibility with enforce** *(review, integrator blocker — net-new).* The shipped OAuth/Plaid marks write into the **user's own existing space** (`channel = authCell.getAsNormalizedFullLink().space`, `oauth2-common.utils.ts:164`, `plaid-oauth.utils.ts:172`; `as: identity`, `index.ts:25`). These are **incompatible with enforce on those spaces**: under enforce, if the operator's `custodyIngest.set` is the first open of a `serverSeq===0` user space it seeds `{ [operator]: OWNER }` (cross-tenant capture, `server.ts:837-849`); if the space exists with no ACL doc the operator write is denied (`server.ts:725-732`). **Default:** the per-space enforce set NEVER includes any space carrying operator-`as:identity` writes — OAuth/Plaid user spaces and existing webhook stream cells **stay `mode:off`**. State this as a hard scoping invariant. The OAuth code already anticipates migration to dedicated channel spaces "once the owner grant lands" (`oauth2-common.utils.ts:155-159`); that migration is a **follow-up**, not this PR — but the invariant that these spaces never enter the enforce set must ship now.

## 10. How the review changed this plan

**Blockers — resolved:**

(A fail-closed stopgap lands immediately; see commit history.)

3. **Seeding-race authority inversion — operator AND presenter** (4 lenses). **Changed:** Step 0 is a tested provisioning state machine; Step 1b adds a **server-side seeding guard** + endpoint refusal to ingest to an unseeded space.
4. **OAuth/Plaid user-space ↔ enforce incompatibility** (integrator lens). **Changed:** §9 risk 9 — hard scoping invariant that the enforce set never includes operator-`as:identity` spaces.
5. **Create/grant UI architecturally infeasible in `cf-webhook`** (completeness lens). **Changed:** Step 2 moves the grant to a runtime-bearing surface (CLI-proven); `cf-webhook` becomes a thin presenter.
6. **Owner-authors / operator-writes chicken-and-egg under enforce** (completeness lens). **Changed:** Step 0 sequences enforce-membership → owner seed → presenter+operator grant → release secret, as a tested order.
